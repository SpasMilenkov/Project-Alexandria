## Architecture Overview

### Backend Services

Alexandria's backend follows a **microservices pattern** with dedicated workers:

- **API Service** — Main HTTP REST API handling requests from the frontend
- **Document Worker** — Processes document uploads (PDF extraction, format conversion via LibreOffice/Tika)
- **Media Worker** — Handles media processing (thumbnails, video previews, FFMPEG operations)

These services communicate via **RabbitMQ** message queues for async task processing.

### Frontend Application

The frontend is a **Single Page Application (SPA)** built with:

- **Vue 3** with Composition API
- **Vite** for fast development and optimized builds
- **Pinia** for state management
- **Vue Router** for navigation

### Data Layer

- **PostgreSQL** — Primary database with EF Core ORM
- **MinIO** — S3-compatible object storage for files
- **RabbitMQ** — Message broker for background jobs

## Key Directories Explained

### `/Backend/API`

The main API service exposing RESTful endpoints. Uses **feature-based organization** where each feature (Files, Auth, Search, etc.) contains its controllers, services, and logic in dedicated folders.

### `/Backend/DocumentWorker.Service` & `/Backend/MediaWorkerService`

Background workers that consume messages from RabbitMQ queues. These handle CPU-intensive operations like:

- Text extraction from PDFs/DOCX
- Document format conversion
- Image thumbnail generation
- Video preview generation

### `/Backend/Data`

Contains the **EF Core DbContext**, entity configurations, and database migrations. This is where the database schema is defined and managed.

### `/Backend/Models`

Domain entities representing the core data structures:

- `File`, `Directory`, `Tag`
- `ApplicationUser`, `ApplicationRole`
- `AuditLog`, `Preview`, `SignedUrl`

### `/Backend/Repositories`

Data access layer implementing the **Repository Pattern**. Each repository handles database operations for specific entities with optimized queries and projections.

### `/Backend/Storage`

Handles all file storage operations through **MinIO** (S3-compatible object storage). Includes upload/download streaming, multipart uploads, and cleanup services.

### `/Frontend/src`

| Directory      | Purpose                                               |
| -------------- | ----------------------------------------------------- |
| `views/`       | Full page components (Browse, Upload, Settings, etc.) |
| `components/`  | Reusable UI components                                |
| `stores/`      | Pinia stores for global state (auth, files, tags)     |
| `api/`         | Axios HTTP client with JWT interceptors               |
| `composables/` | Reusable composition functions                        |
| `router/`      | Route definitions and navigation guards               |

## Development vs Production

The project includes separate Docker Compose configurations:

- **`docker-compose.dev.yml`** — Development setup with hot reload, exposed ports, and debug configurations
- **`docker-compose.yml`** — Production-optimized with Nginx reverse proxy and secure defaults
